# 業務ルール

本書は `phone-order-api` が前提とする注文管理上の業務ルールを整理する。  
現行実装で確認できる制約を対象とする。

---

## 概要

- 注文は注文コードで一意に識別する
- 注文登録時には注文日時が必須である
- 注文登録直後の状態は受付である
- 注文キャンセルには状態制約とバージョン一致が必要である

---

## 内容

### ルール

#### 注文識別

- 外部向けの注文識別子は `orderCode` とする
- `orderCode` は `ORD` + 6 桁の数値で表現する
- `orderCode` はシステム側で採番する

#### 注文登録

- 注文登録時には注文日時を必須とする
- 注文登録時の注文状態は `001:受付` とする
- 注文登録時のバージョンは `0` とする

#### 注文状態

注文状態は次のコードで管理する。

| コード   | 名称    |
|-------|-------|
| `001` | 受付    |
| `002` | 審査中   |
| `003` | 手配中   |
| `004` | 出荷待ち  |
| `005` | 完了    |
| `006` | キャンセル |

#### キャンセル

- 注文キャンセル時には要求バージョンを指定する
- 要求バージョンと現在のバージョンが一致しない場合はキャンセルできない
- 注文状態が `005:完了` の場合はキャンセルできない
- 注文状態が `006:キャンセル` の場合はキャンセルできない
- 上記以外の状態ではキャンセル後の状態を `006:キャンセル` とする

#### 一覧取得

- 注文一覧は注文日時の降順で取得する
- 注文日時が同一の場合は注文コードの降順で取得する
- ページ番号は `0` 始まりとする
- ページサイズのデフォルトは `20` とする
- ページサイズは `1` 以上 `100` 以下とする
- 範囲外のページを指定した場合も空ページとして返す

#### エラー

- 入力値不正時は `400 Bad Request` を返す
- 指定された注文が存在しない場合は `404 Not Found` を返す
- キャンセル不可またはバージョン競合時は `409 Conflict` を返す
- 想定外エラー時は `500 Internal Server Error` を返す
