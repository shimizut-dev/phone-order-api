# システム概要

本書は `phone-order-api` の機能と構成を要件レベルで整理する。  
現行コードと API 契約から確認できる範囲に限定する。

---

## 概要

- `phone-order-api` は移動機注文を管理する REST API である
- API は JSON を利用し、外部公開パスは `/api/v1` を先頭に付ける
- データベースには PostgreSQL を利用する
- 注文データは注文コード、注文日時、注文状態、バージョンを中心に管理する

---

## 内容

### 提供機能

| 機能      | 概要                         |
|---------|----------------------------|
| 注文一覧取得  | 注文日時降順で注文をページング取得する        |
| 注文取得    | 注文コードを指定して 1 件取得する         |
| 注文登録    | 注文日時を指定して注文を登録する           |
| 注文キャンセル | 注文コードとバージョンを指定して注文をキャンセルする |

### 公開 API

| メソッド   | パス                                  | 用途      |
|--------|-------------------------------------|---------|
| `GET`  | `/api/v1/orders`                    | 注文一覧取得  |
| `POST` | `/api/v1/orders`                    | 注文登録    |
| `GET`  | `/api/v1/orders/{orderCode}`        | 注文取得    |
| `POST` | `/api/v1/orders/{orderCode}/cancel` | 注文キャンセル |

### 入出力の前提

- リクエストとレスポンスは JSON を利用する
- 日時はタイムゾーンオフセット付き ISO 8601 形式で扱う
- `orderCode` は `ORD` + 6 桁の形式とする
- エラー時は共通の `ErrorResponse` を返す

### 管理データ

現行実装で主要に扱う注文データは次のとおり。

- `orderCode`
- `orderedAt`
- `orderStatus`
- `version`
- `createdAt` / `createdBy`
- `updatedAt` / `updatedBy`

### 技術上の前提

- Spring Boot を利用する
- JPA と Flyway で永続化を管理する
- OpenAPI を API 仕様の正本とする
- 楽観的ロックにより注文更新競合を検知する
