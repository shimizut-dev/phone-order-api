# エラーレスポンス

本書は `phone-order-api` のエラーレスポンス方針を整理する。  
具体的な JSON 構造と example は `OpenAPI仕様.yaml` を正本とする。

---

## 概要

- HTTP ステータスの使い分けを統一する
- 共通エラーレスポンス形式を整理する
- presentation 層での例外変換方針を明確にする

---

## 内容

### 方針

- エラー時は共通の `ErrorResponse` を返す
- バリデーションエラー詳細は `validationErrors` に `ValidationError` 配列で返す
- `validationErrors` は該当がない場合も空配列を返す
- 業務ルール違反と入力不正は別の HTTP ステータスで表現する

### ルール

| ステータス | 用途            | 主な発生例                        |
|-------|---------------|------------------------------|
| `400` | リクエスト不正       | JSON 形式不正、注文コード形式不正、監査ユーザー不正 |
| `404` | リソース未存在       | 注文未存在                        |
| `409` | 業務ルール違反、状態不整合 | キャンセル不可、バージョン競合              |
| `500` | 想定外エラー        | 永続化済みデータ不整合、未捕捉例外            |

- `ApiExceptionHandler` で例外を HTTP エラーへ変換する
- `OrderNotFoundException` は `404` に変換する
- `OrderCannotBeCancelledException` は `409` に変換する
- `OrderVersionConflictException` と `ObjectOptimisticLockingFailureException` は `409` に変換する
- `ApiBadRequestException`、`MethodArgumentNotValidException`、`ConstraintViolationException`、`HttpMessageNotReadableException`、`InvalidAuditorException` は `400` に変換する
- `InvalidPersistedOrderException` は `500` に変換する
- `error` フィールドには標準 HTTP ステータス名、または `HTTP_<statusCode>` を返す

### 手順

1. 新しい例外を追加する場合は、どのレイヤーで送出し、どの HTTP ステータスへ変換するかを決める
2. `ApiExceptionHandler` に変換ルールを追加する
3. `OpenAPI仕様.yaml` の該当 API に発生し得るレスポンスだけを追加する
4. `ErrorResponse` と `ValidationError` の構造を変える場合は、生成モデルと実装双方の影響を確認する
