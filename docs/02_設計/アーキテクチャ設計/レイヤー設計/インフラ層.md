# インフラ層

本書は `phone-order-api` のインフラ層を整理する。  
現行の JPA、採番、監査設定、ログ基盤実装を基準とする。

---

## 概要

- domain 境界の実装を提供する
- DB 永続化、採番、監査、ログなどの技術的詳細を扱う
- JPA と PostgreSQL を前提にした実装を閉じ込める

---

## 内容

### 方針

- domain と application が必要とする契約だけを実装する
- 永続化表現と domain 表現の差分は mapper で吸収する
- フレームワーク依存はインフラ層で完結させる

### ルール

- `OrderRepositoryImpl` は `OrderRepository` を実装する
- `OrderJpaRepository` は Spring Data JPA の repository として `orders` テーブルへアクセスする
- `OrderJpaEntity` は `orders` テーブルの永続化表現を持つ
- `OrderJpaMapper` は JPA entity と domain `Order` の相互変換を担う
- `SequenceOrderCodeGenerator` は `OrderCodeGenerator` を実装し、`next_order_code()` を呼び出して採番する
- `JpaAuditConfig` は `X-User-Id`、`X-User-Name`、`X-Actor` の順で監査ユーザーを解決する
- `BaseAuditJpaEntity` は `created_at`、`created_by`、`updated_at`、`updated_by`、`deleted_at`、`deleted_by` を共通監査カラムとして持つ
- `MethodLogAspect` と `LogMasker` はログ出力の技術基盤として扱う
- HTTP レスポンス形式や業務ルール判断はインフラ層に持ち込まない

### 手順

1. 新しい永続化対象を追加する場合は、JPA entity、mapper、repository 実装の責務を分ける
2. domain interface を先に定義し、その後で infrastructure 実装を追加する
3. DB 変更が必要な場合は Flyway migration を追加し、ER 図と DB 設計資料も更新する
