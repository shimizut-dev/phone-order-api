# ドメイン層

本書は `phone-order-api` のドメイン層を整理する。  
現行の `domain.common` と `domain.order` を基準とする。

---

## 概要

- 注文に関する業務ルールと不変条件を表現する
- `Order` 集約、値オブジェクト、例外、repository interface を定義する
- フレームワークや永続化方式から独立させる

---

## 内容

### 方針

- 不変条件は値オブジェクトまたは集約の振る舞いで保証する
- 外部 I/O やフレームワークの詳細は domain に持ち込まない
- ドメイン境界は interface で表現する

### ルール

- `Order` は現行ドメインの中心集約である
- `OrderId`, `OrderCode`, `OrderedAt`, `Version` は値オブジェクトとして制約を持つ
- `OrderStatus` は定義済みコードのみを許可する
- `Order.cancel` はバージョン一致を確認し、完了済みまたはキャンセル済みの注文を拒否する
- `OrderRepository` は注文取得、登録、更新の契約を定義する
- `OrderCodeGenerator` は注文コード採番の契約を定義する
- `PageResult` と `PagingCondition` はページング付き一覧取得の共通表現とする
- Spring、Servlet、JPA のアノテーションや型へ依存しない

### 手順

1. 新しい業務ルールを追加する場合は、まず値オブジェクトで閉じるか集約の振る舞いで扱うかを判断する
2. 技術的詳細が必要な場合は interface だけを domain に置く
3. ルール追加時は対応するドメインテストも追加する
