# 採用方針

本書は `phone-order-api` で採用するアーキテクチャ方針を整理する。  
現行コードで実際に使っている設計と技術要素を基準とする。

---

## 概要

- DDD と Clean Architecture の考え方を土台にする
- ただし、現行スコープに対して必要な範囲に絞って適用する
- 設計上の原則と実装上の現実のずれを明示して運用する

---

## 内容

### 方針

- ドメイン層では業務ルールと不変条件を表現する
- アプリケーション層ではユースケース実行とトランザクション境界を扱う
- プレゼンテーション層では HTTP 入出力と例外変換を扱う
- インフラ層では JPA、採番、監査、ログなどの技術的詳細を扱う
- スコープが小さい間は層内のクラス数を必要以上に増やさない

### ルール

- ドメインモデルはフレームワークに依存しない
- OpenAPI 生成モデルや JPA エンティティを domain に持ち込まない
- repository interface は domain に置き、その実装は infrastructure に置く
- 採番は `OrderCodeGenerator` で抽象化し、実装は `SequenceOrderCodeGenerator` に置く
- 現行実装で専用の domain service は置かず、必要なルールは `Order` と値オブジェクトへ寄せる
- 依存ルールは `ArchitectureTest` で一部自動検証する

### 手順

1. 新しい責務を追加する場合は、まず domain へ置くべきかを判断する
2. domain に置けない技術的詳細は infrastructure へ寄せる
3. 外部入力表現と内部モデルの差分は presentation の mapper で吸収する
4. 設計原則を破る必要がある場合は、理由と影響範囲を明示する
