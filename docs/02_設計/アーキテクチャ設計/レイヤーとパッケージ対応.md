# レイヤーとパッケージ対応

本書は設計上のレイヤー用語と、`phone-order-api` の実装パッケージの対応を整理する。  
設計説明と実装説明の呼び方を揃えることを目的とする。

---

## 概要

- 資料上のレイヤー用語と Java パッケージの対応を明確にする
- 用語のぶれを減らし、実装位置を判断しやすくする
- OpenAPI 生成物の扱いも合わせて整理する

---

## 内容

### 方針

- 資料上では `プレゼンテーション層`、`アプリケーション層`、`ドメイン層`、`インフラ層` を基本用語とする
- 実装説明では必要に応じてパッケージ名を併記する
- 同じ層に複数の別名を増やさない

### ルール

| 設計上の用語     | 実装パッケージ          | 主な責務                                            |
|------------|------------------|-------------------------------------------------|
| プレゼンテーション層 | `presentation`   | HTTP 入出力、OpenAPI 生成モデルとの接続、例外変換、フィルタ            |
| アプリケーション層  | `application`    | ユースケース実行、トランザクション制御、ドメイン境界の調整                   |
| ドメイン層      | `domain`         | 集約、値オブジェクト、例外、repository interface、採番 interface |
| インフラ層      | `infrastructure` | JPA、repository 実装、採番実装、監査設定、ログ基盤                |

- `presentation.generated` は OpenAPI 仕様から生成されるプレゼンテーション層の一部として扱う
- `infrastructure.persistence` は DB 永続化の実装詳細を置く
- `infrastructure.repository` は domain repository の実装を置く

### 手順

1. 資料を書くときはレイヤー用語を使い、必要に応じてパッケージ名を併記する
2. 実装を読むときは、まず該当レイヤのトップパッケージを確認する
3. 生成物を説明する場合は `presentation.generated` であることを明記する
