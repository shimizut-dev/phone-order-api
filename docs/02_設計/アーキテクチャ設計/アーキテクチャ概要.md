# アーキテクチャ概要

本書は `phone-order-api` のアーキテクチャ全体像を整理する。  
現行実装のレイヤ構成、責務分担、主要な技術要素を基準とする。

---

## 概要

- `phone-order-api` は移動機注文を管理する REST API である
- レイヤ構成は `presentation`、`application`、`domain`、`infrastructure` の 4 層で整理する
- 外部 I/O の詳細を外側へ寄せ、業務ルールを内側に閉じ込める

---

## 内容

### 方針

- ドメインの整合性を中心に据える
- HTTP、JPA、採番、ログなどの技術的詳細は外側の層へ寄せる
- 実装はシンプルな層分割を維持し、過度な抽象化は避ける

### ルール

- レイヤは `presentation -> application -> domain` の依存方向を基本とする
- `infrastructure` は `domain` の境界と `application` の要求を実装する
- 現行ユースケースは注文一覧取得、注文取得、注文登録、注文キャンセルの 4 件である
- 中心となるドメインは `Order` 集約である
- OpenAPI 生成物は `presentation.generated` 配下へ出力され、controller 実装はそれに従う
- DB アクセスは JPA と Flyway を利用する

### 手順

1. 新しい機能を追加する場合は、まずどのレイヤに責務を置くかを判断する
2. 外部入力は presentation で受け、application を経由して domain へ渡す
3. 永続化や採番が必要な場合は domain 境界を定義し、infrastructure で実装する
