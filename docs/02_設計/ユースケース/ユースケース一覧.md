# ユースケース一覧

本書は `phone-order-api` の主要なユースケースを整理する。  
現行コードと API 契約で確認できる操作を対象とする。

---

## 概要

- 現行実装の主要ユースケースは注文一覧取得、注文取得、注文登録、注文キャンセルである
- 利用者は API を通じて注文の参照と状態変更を行う
- ユースケースごとに入力、主処理、主な結果を整理する

---

## 内容

### 方針

- 現行コードと OpenAPI で確認できるユースケースだけを扱う
- 画面操作ではなく API 利用を前提とした流れで整理する
- 業務ルールの詳細は `業務ルール.md` を正本として参照する

### ルール

- 注文の外部向け識別子は `orderCode` とする
- 注文一覧は注文日時降順、同一日時は注文コード降順で取得する
- 注文登録時の初期状態は `001:受付` とする
- 注文キャンセル時にはバージョン一致が必要である
- 完了済みまたはキャンセル済みの注文はキャンセルできない

### ユースケース

#### 注文一覧を取得する

- 利用者
  - 受注担当者
  - 業務管理者
- 入力
  - `page`
  - `size`
- 主処理
  1. 利用者は注文一覧取得を要求する
  2. システムはページング条件を補完または検証する
  3. システムは注文日時降順で注文一覧を取得する
  4. システムはページ情報付きで注文一覧を返す
- 主な結果
  - 指定条件の注文一覧を返す
  - 範囲外ページの場合は空ページを返す

#### 注文を取得する

- 利用者
  - 受注担当者
  - 業務管理者
- 入力
  - `orderCode`
- 主処理
  1. 利用者は注文コードを指定して注文取得を要求する
  2. システムは注文コード形式を検証する
  3. システムは注文コードに対応する注文を取得する
  4. システムは注文情報を返す
- 主な結果
  - 対象の注文を 1 件返す
  - 対象が存在しない場合は `404 Not Found` を返す

#### 注文を登録する

- 利用者
  - 受注担当者
- 入力
  - `orderedAt`
- 主処理
  1. 利用者は注文日時を指定して注文登録を要求する
  2. システムは入力値を検証する
  3. システムは注文コードを採番する
  4. システムは受付状態の注文を登録する
  5. システムは登録結果を返す
- 主な結果
  - 注文コード付きの注文を返す
  - 登録直後の状態は `001:受付` である
  - 登録直後のバージョンは `0` である

#### 注文をキャンセルする

- 利用者
  - 受注担当者
  - 業務管理者
- 入力
  - `orderCode`
  - `version`
- 主処理
  1. 利用者は注文コードとバージョンを指定してキャンセルを要求する
  2. システムは注文コード形式とバージョンを検証する
  3. システムは対象注文を取得する
  4. システムは状態とバージョンを検証する
  5. システムは注文をキャンセル状態に更新する
  6. システムは更新後の注文情報を返す
- 主な結果
  - 注文状態を `006:キャンセル` に更新して返す
  - 対象が存在しない場合は `404 Not Found` を返す
  - 状態不整合またはバージョン競合時は `409 Conflict` を返す

### 手順

1. 利用者が目的に応じて対象 API を選択する
2. システムは入力値を検証する
3. システムは注文取得または更新処理を実行する
4. システムは成功時に注文情報、失敗時に共通エラーレスポンスを返す
