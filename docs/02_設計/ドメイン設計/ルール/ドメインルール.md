# ドメインルール

本書は `phone-order-api` のドメイン層で守るべき主要ルールを整理する。  
現行の `order` ドメイン実装を基準とする。

---

## 概要

- ドメイン層は注文に関する整合性を担う
- 注文コード形式、注文状態、バージョン整合性はドメインで管理する
- 永続化や採番の具体実装はドメイン境界の外に分離する

---

## 内容

### 方針

- 業務上の不変条件は値オブジェクトまたは集約の振る舞いで保証する
- 例外は不整合や不正入力を明確に表すために使う
- ドメイン層はフレームワーク詳細に依存しない
- 永続化、採番、HTTP 入出力の詳細はドメイン層に持ち込まない

### ドメイン要素一覧

| 要素       | 現行実装                                        | 整理内容                                         |
|----------|---------------------------------------------|----------------------------------------------|
| エンティティ   | `Order`                                     | 同一性を持つ注文本体。集約のルートエンティティでもある                  |
| 値オブジェクト  | `OrderId` `OrderCode` `OrderedAt` `Version` | それぞれの型で入力制約と不変条件を表現する                        |
| ドメインサービス | 専用クラスは未配置                                   | 現状の業務ルールは `Order` と値オブジェクトに閉じている             |
| リポジトリ    | `OrderRepository`                           | 注文の取得、登録、更新を担うドメイン境界                         |
| 採番       | `OrderCodeGenerator`                        | 注文コードの採番規則と実装詳細を隠蔽する境界                       |
| ファクトリ    | `Order.create` `Order.reconstruct`          | 専用ファクトリクラスは置かず、ルートエンティティのファクトリメソッドで生成と再構築を行う |

### エンティティ

- 現行実装のエンティティは `Order` である
- `Order` は `OrderId` によって同一性を持つ
- `Order` は注文全体を表すルートエンティティであり、状態変更の入口になる
- ルートエンティティを経由せずに注文状態を変更しない

### 値オブジェクト

- `OrderId`
  - 注文の内部識別子を表す
- `OrderCode`
  - 外部向け注文識別子を表す
  - `ORD` + 6 桁の形式を保証する
- `OrderedAt`
  - 注文日時を表す
  - null を許可しない
- `Version`
  - 更新競合検知に使うバージョンを表す
  - 0 以上を保証する

### ドメインサービス

- 現行実装では専用のドメインサービスクラスは置いていない
- 現在の業務ルールは `Order` と値オブジェクトで閉じている
- 複数集約にまたがる計算や判定が今後追加され、ルートエンティティに閉じ込めにくい場合に限ってドメインサービスを検討する

### リポジトリ

- `OrderRepository` は注文集約を扱うドメイン境界である
- ドメイン層には取得、登録、更新の契約のみを置く
- SQL、JPA、ページング実装詳細はインフラストラクチャ層に閉じ込める

### 採番

- `OrderCodeGenerator` は注文コード採番を表すドメイン境界である
- ドメイン層は採番の必要性だけを表現し、採番方式自体は持たない
- 連番生成やシーケンス利用などの実装詳細はインフラストラクチャ層で実装する

### ファクトリ

- 現行実装では専用のファクトリクラスは置かない
- 新規生成は `Order.create` を使う
- 永続化済みデータからの再構築は `Order.reconstruct` を使う
- 生成入口をルートエンティティに寄せることで、初期状態の整合性を一か所で管理する

### ルール

- `OrderCode` は `ORD` + 6 桁の形式でなければならない
- `OrderedAt` は null を許可しない
- `Version` は 0 以上でなければならない
- `Order` の主要属性はすべて必須である
- 注文の生成と再構築は `Order` のファクトリメソッドを入口とする
- `Order.cancel` は要求バージョン一致を前提とする
- `Order.cancel` は完了済みまたはキャンセル済みの注文に対して実行できない
- `OrderStatus` は定義済みコードのみ有効とする
- `OrderRepository` はドメイン層の境界として定義し、永続化方式は外側の層に閉じ込める
- `OrderCodeGenerator` は採番の詳細を隠蔽する境界として扱う
- 専用ドメインサービスは、集約や値オブジェクトに責務を置けない場合に限って追加する
- 専用ファクトリクラスは、ルートエンティティのファクトリメソッドで責務を持てない場合に限って追加する

### 手順

1. 新しいドメインルールを追加する場合は、値オブジェクトで閉じるか集約の振る舞いで扱うかを決める
2. エンティティ、値オブジェクト、ドメインサービス、リポジトリ、採番、ファクトリのどこに責務を置くかを先に判断する
3. ルール違反時にどの例外で表現するかを明確にする
4. ルール追加時は対応するテストも追加する
