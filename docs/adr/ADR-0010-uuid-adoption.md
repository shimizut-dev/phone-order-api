# ADR-0010: UUID を採用する

## ステータス

採用

## 日付

2026-04-04

---

## コンテキスト

phone-order-api プロジェクトでは、主キーの生成方式を決定する必要がある。

候補として以下を検討した。

- UUID
- 連番（Auto Increment）
- 業務ID

検討観点

- 一意性
- 分散環境対応
- セキュリティ
- 実務適用性
- 拡張性

---

## 決定

UUID を採用する

---

## 理由

### 1. 一意性の保証

UUID はグローバルに一意な ID を生成できるため、
システム間連携や将来の拡張に対応しやすい。

---

### 2. 分散環境への対応

UUID は DB に依存せずに ID を生成できるため、
分散システムやマイクロサービスにも対応しやすい。

---

### 3. セキュリティ向上

連番 ID の場合、
ID 推測が容易になる。

UUID を使用することで、
推測が困難になる。

---

### 4. 実務での採用が増えている

近年のクラウド環境では UUID の採用が増えている。

---

## 連番（Auto Increment）との比較

### メリット

- 可読性が高い
- インデックス効率が良い
- パフォーマンスが良い

### デメリット

- 分散環境に弱い
- ID 推測が可能
- DB 依存になる

---

## 業務IDとの比較

### メリット

- 人間に分かりやすい
- 業務で使用しやすい

### デメリット

- 一意性管理が必要
- 変更される可能性がある

---

## UUIDのデメリット

- サイズが大きい
- インデックス効率がやや低い
- 可読性が低い

---

## 運用方針

- 主キーは UUID を使用する
- 業務IDは別途用意する
- PostgreSQL の UUID 型を使用する

---

## PostgreSQL 例

```text
id uuid primary key default gen_random_uuid()
```

---

## 影響

### ポジティブな影響

- 一意性向上
- 分散対応
- セキュリティ向上

---

### ネガティブな影響

- インデックスサイズ増加
- 可読性低下

---

## 結論

phone-order-api では UUID を主キーとして採用する。

分散対応・拡張性・セキュリティの観点から最適なため。
