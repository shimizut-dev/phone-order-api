# ADR-0011: REST API 設計方針を採用する

## ステータス

採用

## 日付

2026-04-04

---

## コンテキスト

phone-order-api プロジェクトでは、外部公開および内部利用を想定した Web API の設計方針を決定する必要がある。

候補として以下を検討した。

- REST API
- RPC 形式 API
- GraphQL

検討観点

- 実務での採用状況
- API の分かりやすさ
- 保守性
- 拡張性
- クライアントとの疎結合
- ポートフォリオとしての説明しやすさ

---

## 決定

REST API 設計方針を採用する

---

## 理由

### 1. 業務リソースを自然に表現しやすいため

phone-order-api では以下のような業務リソースを扱う。

- 注文
- 注文明細
- 配送
- 注文関係者

REST はこれらの業務概念をリソースとして表現しやすい。

例

- `/orders`
- `/orders/{orderId}`
- `/deliveries/{deliveryId}`

---

### 2. 実務で広く採用されているため

REST API は Web API の代表的な設計方針であり、
多くのシステムで採用されている。

実務想定のポートフォリオとして説明しやすい。

---

### 3. クライアントとの疎結合を保ちやすいため

HTTP メソッドとリソース指向に基づく設計により、
クライアントとサーバの責務分離がしやすい。

---

### 4. Spring Boot と相性が良いため

Spring Boot は REST API 開発に必要な機能を標準で提供しており、
実装・テスト・例外ハンドリング・バリデーションを統一しやすい。

---

### 5. 将来的な拡張がしやすいため

REST API はエンドポイント単位で責務を整理しやすく、
一覧取得・詳細取得・作成・更新・キャンセルなどを段階的に拡張しやすい。

---

## 設計方針

phone-order-api では以下の REST 設計方針を採用する。

### 1. リソース名は名詞・複数形を基本とする

例

- `/orders`
- `/deliveries`

動詞を URL に含めるのは最小限とする。

---

### 2. HTTP メソッドで操作を表現する

- `GET` : 参照
- `POST` : 作成
- `PUT` : 全体更新
- `PATCH` : 部分更新
- `DELETE` : 削除

ただし、業務上のキャンセルなどは単純削除ではなく状態変更として扱う。

---

### 3. 業務操作はサブリソースまたは明示的アクションとして表現する

例

- `POST /orders`
- `GET /orders/{orderId}`
- `POST /orders/{orderId}/cancel`

業務的に意味のある操作は、
単純な CRUD に無理やり当てはめず、可読性を優先する。

---

### 4. レスポンスは JSON を採用する

標準的なデータ交換形式として JSON を採用する。

---

### 5. エラーは HTTP ステータスと業務エラー情報を併用する

例

- 400 Bad Request
- 404 Not Found
- 409 Conflict
- 422 Unprocessable Entity
- 500 Internal Server Error

アプリケーション独自のエラーコード・メッセージを必要に応じて返却する。

---

### 6. API バージョニングを導入する

例

- `api/v1/orders`

---

### 7. DTO を利用し、ドメインオブジェクトを直接公開しない

API の入出力は Request DTO / Response DTO を使用する。
ドメインモデルと API 契約を分離する。

---

## RPC 形式 API との比較

### RPC のメリット

- 操作中心で分かりやすい場合がある
- 複雑な業務操作を表現しやすい

### RPC のデメリット

- URL が動詞中心になりやすい
- リソースの一貫性が崩れやすい
- API 全体の統一感を保ちにくい

---

## GraphQL との比較

### GraphQL のメリット

- 必要な項目だけ取得できる
- クライアント柔軟性が高い

### GraphQL のデメリット

- 学習コストが高い
- MVP には過剰
- キャッシュや認可設計が複雑になりやすい

---

## 影響

### ポジティブな影響

- API 設計の統一
- 実務で説明しやすい
- Spring Boot と相性が良い
- フロントや外部利用者に理解されやすい

### ネガティブな影響

- 業務操作によっては REST にきれいに当てはめにくい
- 厳密な REST にこだわりすぎると可読性が下がる場合がある

---

## 補足方針

- REST を基本とするが、業務上不自然な設計は避ける
- 「純粋な REST」よりも「実務で分かりやすい API」を優先する
- リソース指向と業務可読性のバランスを取る

---

## 結論

phone-order-api では REST API 設計方針を採用する。

業務リソースを自然に表現しやすく、
実務想定・保守性・説明しやすさの観点で最も適しているため。
