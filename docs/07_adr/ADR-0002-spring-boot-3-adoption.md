# ADR-0002: Spring Boot 3 を採用する

## ステータス

採用

## 日付

2026-04-04

---

## コンテキスト

phone-order-api プロジェクトでは、Java / Spring ベースで Web API を開発する。

フレームワーク選定にあたり、以下の観点を重視した。

- Web API 開発の生産性
- 実務での採用実績
- 学習価値
- 保守性
- Java 21 との相性
- 今後の拡張性

候補として主に以下を検討した。

- Spring Framework をベースに必要機能を個別構成する
- Spring Boot 3 を採用する

---

## 決定

Spring Boot 3 を採用する。

---

## 理由

### 1. Web API 開発の生産性が高いため

Spring Boot は、依存関係管理・自動設定・アプリ起動・設定管理などが標準で整っており、
Web API 開発をスムーズに開始できる。

これにより、フレームワークの配線よりも業務実装に集中しやすい。

### 2. 実務での利用実績が高いため

Spring Boot は Java の Web API 開発で広く利用されており、
実務想定のポートフォリオとしても説明しやすい。

### 3. Java 21 と相性が良いため

本プロジェクトでは Java 21 を採用しており、
Spring Boot 3 は Java 17 以上を前提とするため整合性が高い。

### 4. Spring Framework 6 系ベースであり将来性が高いため

Spring Boot 3 は Spring Framework 6 系をベースとしており、
Jakarta EE 対応を含め、今後の標準的な構成に乗りやすい。

### 5. 周辺機能との統合がしやすいため

以下のような構成を導入しやすい。

- Spring Web
- Validation
- Spring Data JPA
- Flyway
- Testcontainers
- Actuator

phone-order-api で想定している開発スタイルと相性が良い。

### 6. DDD / レイヤードアーキテクチャとの両立がしやすいため

Spring Boot は便利機能が多い一方で、
アプリケーション層・ドメイン層・インフラ層の分離を意識した構成も取りやすい。

そのため、実務想定と DDD ポートフォリオの両立に向いている。

---

## 影響

### ポジティブな影響

- 初期構築を早く進められる
- 実務で説明しやすい技術選定になる
- Java 21 と組み合わせやすい
- 将来的なライブラリ追加がしやすい
- テストや運用系の仕組みを載せやすい

### ネガティブな影響

- Spring に依存した設計へ寄りやすい
- 自動設定に頼りすぎると内部理解が浅くなりやすい
- 小規模プロジェクトでは構成がやや重く感じる場合がある

---

## 代替案

### 1. Spring Framework を直接利用する

メリット

- 構成を細かく制御できる
- 仕組みの理解が深まりやすい

デメリット

- 初期構築コストが高い
- 学習目的としては良いが、実装速度が落ちやすい
- phone-order-api の目的に対しては過剰になりやすい

### 2. 他の Java Web フレームワークを採用する

メリット

- 軽量性や起動速度など、別の強みを得られる場合がある

デメリット

- 今回の学習テーマ、実務想定、既存知識との一貫性が弱くなる
- Spring 系技術との組み合わせ前提で考えている構成とずれる

---

## 補足方針

Spring Boot 3 を採用するが、以下の方針を併用する。

- ドメイン層に Spring 依存を極力持ち込まない
- 自動設定に頼りすぎず、必要箇所は明示的に理解する
- 業務ロジックはフレームワーク都合ではなくドメイン都合で設計する

---

## 結論

phone-order-api では Spring Boot 3 を採用する。
実務想定の開発生産性と、DDD ポートフォリオとしての説明しやすさを両立できるためである。
