# エラーレスポンス

## 1. 位置づけ

エラーレスポンスの具体的な JSON 構造と example は `openapi.yaml` を正本とする。
本ドキュメントでは、HTTP ステータスの使い分けと運用方針を補足する。

---

## 2. 基本方針

- エラー形式を統一する
- HTTP ステータスを適切に使い分ける
- バリデーションエラーと業務ルール違反を区別する
- 想定外エラーは共通形式で返却する
- API 利用者向けのメッセージを返却する

---

## 3. HTTP ステータス方針

| ステータス | 用途              |
|-------|-----------------|
| 400   | リクエスト形式不正、入力値不正 |
| 404   | 対象リソースが存在しない    |
| 409   | 業務ルール違反、状態不整合   |
| 500   | 想定外エラー          |

---

## 4. ステータス別ルール

### 4.1 400 Bad Request

以下の場合に使用する。

- JSON 構造不正
- 必須項目不足
- 型不正
- 形式不正
- presentation 層で判定できる入力不正

### 4.2 404 Not Found

以下の場合に使用する。

- 指定した注文が存在しない

### 4.3 409 Conflict

以下の場合に使用する。

- 注文キャンセル不可状態
- ドメインルール違反
- 業務上ありえない状態

### 4.4 500 Internal Server Error

以下の場合に使用する。

- 想定外例外
- 実装不備
- 一時的な内部障害

---

## 5. レイヤ別責務

### 5.1 presentation

- 入力形式エラーを 400 に変換する
- domain / application の例外を HTTP エラーに変換する
- 共通エラーレスポンス形式に整形する

### 5.2 application

- ユースケース実行中の例外を適切に伝搬する
- HTTP には依存しない

### 5.3 domain

- 業務ルール違反を表す例外を送出する
- HTTP ステータスを知らない

---

## 6. OpenAPI との関係

OpenAPI では各 API に発生し得る HTTP ステータスだけを定義する。

- 400: リクエスト形式不正、入力値不正が発生し得る API
- 404: 対象リソースが存在しない可能性がある API
- 409: 業務ルール違反、状態不整合が発生し得る API。現時点では注文キャンセル API で使用する
- 500: 想定外エラーを共通形式で返す API

共通エラースキーマは `openapi.yaml` の `components/schemas/ErrorResponse` と `components/schemas/ValidationError` に定義する。
共通レスポンスは `components/responses` に定義する。
各 example は、レンダラー互換性を優先して `components/responses` の media type に直接定義する。

---

## 7. MVP 時点の簡略方針

MVP では以下を優先する。

- シンプルなエラー形式
- 必要最小限の HTTP ステータス表現
- 一貫した HTTP ステータス

MVP では国際化、多言語メッセージ、複雑なサブコード体系は扱わない。

## 実装上の命名

- 公開 API のエラースキーマ名は `ErrorResponse` / `ValidationError` とする
- 実装側で例外ハンドラ専用の補助 DTO を持つ場合は、OpenAPI 生成モデルと区別するため `ApiErrorResponse` /
  `ApiValidationError` を使う
- `validationErrors` は常に配列で返し、該当なしの場合は `[]` を返す
- `message` は常に文字列で返し、個別メッセージを決められない場合も汎用メッセージへフォールバックする
- `error` は標準ステータスでは `BAD_REQUEST` などを返し、非標準ステータスでは `HTTP_<statusCode>` を返す
