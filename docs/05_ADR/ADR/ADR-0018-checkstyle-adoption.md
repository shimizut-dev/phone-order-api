# ADR-0018: Checkstyle を採用する

## ステータス

採用

## 日付

2026-05-06

---

## コンテキスト

phone-order-api では Java コードとテストコードを継続的に保守する。

命名、import、改行、基本的な記述規約が揃っていないと、
レビューで人手確認が増え、本質的でない差分が積み上がりやすい。

このため、コーディング規約の一部をビルド時に自動検証する仕組みが必要である。

---

## 決定

Checkstyle を採用する。

`maven-checkstyle-plugin` を `validate` フェーズで実行し、
`config/checkstyle/checkstyle.xml` を正本として main / test の両方を検査する。

違反がある場合はビルドを失敗させる。

---

## 理由

- コーディング規約を機械的に検証できる
- `validate` フェーズで早く失敗できるため、無駄な後続処理を減らせる
- main だけでなく test も同じ規律で管理できる
- ルールファイルをリポジトリ管理することで、開発者ごとの差異を減らせる

---

## 代替案

- IDE のコードスタイル設定だけで運用する
- レビュー時の目視確認だけで規約を守る

---

## メリット

- 命名や基本スタイルのばらつきを抑えられる
- レビューで扱うべき論点を設計や振る舞いに寄せられる
- CI で一貫した規約チェックが行える

---

## デメリット

- ルールが厳しすぎると開発速度を下げる可能性がある
- 規約例外が必要な場合は設定調整の手間が発生する
