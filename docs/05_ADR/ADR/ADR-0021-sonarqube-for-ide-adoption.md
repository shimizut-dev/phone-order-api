# ADR-0021: SonarQube for IDE を採用する

## ステータス

採用

## 日付

2026-05-06

---

## コンテキスト

phone-order-api では、Checkstyle や SpotBugs に加えて、
IDE 上で早いフィードバックを得られる静的解析も必要である。

特に実装中にコードスメルや改善候補を確認できると、
CI 実行前に修正でき、手戻りを減らしやすい。

現状の開発環境資料では IntelliJ の追加プラグインとして
`SonarQube for IDE` を利用する前提になっている。

---

## 決定

IntelliJ の開発補助として SonarQube for IDE を採用する。

本採用は IDE 上のローカル解析を対象とし、
Maven ビルドや CI の必須ゲートとしては扱わない。

品質ゲートの正本は引き続き Maven と CI に置き、
SonarQube for IDE は補助的な早期検出手段として運用する。

---

## 理由

- 実装中に静的解析結果を確認でき、修正の往復を減らせる
- Checkstyle や SpotBugs とは異なる観点の改善候補を得られる
- 開発者体験を大きく崩さずに導入できる
- 既存の開発環境資料と整合する

---

## 代替案

- IDE プラグインを使わず、Maven / CI の結果だけで品質確認する
- SonarQube サーバー連携を前提に、CI 側の必須ゲートとして導入する

---

## メリット

- ローカルで早く問題に気づける
- 修正コストが小さい段階で改善しやすい
- IDE 上の補助として導入でき、既存ビルド構成への影響が小さい

---

## デメリット

- IntelliJ 利用前提が強まる
- ビルド必須ゲートではないため、全員が同じ指摘を必ず見るわけではない
- ルールの最終的な統制は Maven / CI 側と分散する
