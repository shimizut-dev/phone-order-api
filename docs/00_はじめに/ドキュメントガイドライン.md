# ドキュメントガイドライン

`docs/` 配下で管理するドキュメントの基本方針を定義します。  
新規作成時は配置場所、命名、テンプレート、文字コードをこのガイドに合わせます。

---

## 概要

- `docs/` は要件、設計、開発、ADR を整理して管理する
- 利用者が迷わないように README から階層的に辿れる構成にする
- Markdown は読み物として整理し、実装や仕様の正本との対応を明確にする
- 文字コードと改行コードはリポジトリ全体で統一する

---

## 内容

### 方針

- `00_はじめに` は `docs/` 全体の入口を置く
- `01_要件` は業務要件、スコープ、関係者、業務ルールを置く
- `02_設計` は API、アーキテクチャ、ドメイン、DB、ユースケースの設計資料を置く
- `03_開発` は開発フロー、環境構築、IDE 設定、コーディング規約を置く
- `05_ADR` は設計判断の記録を置く

### ルール

- 各階層の入口には `README.md` を置く
- 規約・方針系は内容が伝わる日本語ファイル名を使う
- 図や補助資料は用途が分かるディレクトリ配下に置く
- API 仕様の正本は `specs/openapi/openapi.yaml` とする
- `docs/` は仕様の説明や判断理由を残す場所であり、仕様ファイルの重複管理はしない
- 本リポジトリのテキストファイルは UTF-8 に統一する
- UTF-8 BOM は使用しない
- 改行コードや基本的な整形ルールは [.editorconfig](../../.editorconfig) を正本とする


### 手順

1. 追加・変更する資料の配置先を決める
2. 同階層の `README.md` から辿れるようにする
3. 正本ファイルとの関係を明記する
4. UTF-8 / BOM なしで保存する
5. 関連資料へのリンクを更新する
